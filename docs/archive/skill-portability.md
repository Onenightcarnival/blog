# Skill 的可移植性到底取决于什么

skill 的可移植性，看的是文件复制过去之后，效果还能不能跟着过去。

文件复制过去，只是分发成功。麻烦的是另一件事：换一个用户、换一个 agent 产品、换一个运行环境，skill 还能不能做出差不多的结果。

我更关心的是这种业务效果的可移植性。

## Skill 要按工程项目看

现在很多人把 skill 当成一段提示词，或者一个给 agent 看的说明文档。这个理解太轻了。

一个认真写的 skill，更像一个小软件项目。它有入口，有依赖，有运行时，有测试方式，也有自己的文件结构。尤其是 Python skill，不能只写一个 `SKILL.md`，再丢几个脚本就算完。它应该按 Python 工程来管理：依赖写清楚，版本锁住，命令能复现，脚本能单独运行。

`SKILL.md` 当然重要。它告诉 agent 什么时候用这个 skill、怎么用这个 skill、遇到不同情况该走哪条路径。但它只解决“读懂任务”的问题，不解决“把事情做成”的问题。

实际干活的通常是旁边的脚本、模板、示例、测试数据、二进制工具和第三方依赖。一个做 PPT 的 skill，可能要调用渲染脚本、字体、母版和检查工具。一个做数据处理的 skill，可能要依赖 pandas、openpyxl、duckdb。一个做浏览器自动化的 skill，还要处理浏览器版本、驱动、网络和沙箱权限。

这些东西拼在一起，skill 就已经是一个可执行工程。

如果不按工程项目管理，skill 很快会变成一包“在我机器上能跑”的材料。作者知道怎么跑，换个人就只能猜。

## 可移植性有三个变量

一个 skill 从作者机器搬到另一个用户那里，最后业务效果能不能保持，大概取决于三个东西：模型、agent harness、运行环境。

模型决定理解和生成能力。

同一份 `SKILL.md`，给不同模型读，效果不会一样。强模型能补齐很多隐含步骤，弱模型可能连触发时机都判断不稳。模型还会影响风格、推理深度、工具调用的耐心，以及遇到异常时会不会自己绕路。

agent harness 决定执行方式。

这里的 harness 指包在模型外面的那层程序：它怎么发现 skill，怎么把 skill 内容塞进上下文，怎么暴露工具，怎么处理文件权限，怎么管理长任务，怎么把工具结果回传给模型。不同产品的 harness 差异很大。它们就算都支持 skill，实际执行路径也可能完全不同。

运行环境决定代码能不能跑。

同一个 Python 脚本，在 macOS、本地 Linux、Docker、云端沙箱里会遇到不同问题。Python 版本、系统库、字体、浏览器、权限、网络、文件系统路径，任何一个细节不一致，都可能让 skill 的结果变样。

## 只有运行环境能相对确定

这三个变量里，运行环境最能控制。

模型不在 skill 作者手里。用户可能用 GPT-5，也可能用另一个模型。模型版本会升级，能力会变化，工具调用策略也会变。

agent harness 也不在 skill 作者手里。不同客户端、不同 agent 框架、不同 IDE 插件，对 skill 的加载和执行都有自己的实现。作者很难要求所有 harness 按同一种方式工作。

运行环境可以被工程化。依赖、版本、命令、输入输出、系统工具，这些都能写进项目里，也能被验证。

Python skill 至少应该做到：

- 用 `pyproject.toml` 声明项目和依赖
- 在项目配置里声明需要的 Python 版本
- 用 `uv.lock` 锁住依赖版本
- 提供固定命令，比如 `uv run python scripts/check.py`
- 把输入输出路径、环境变量、权限要求写清楚
- 把外部资源放进 skill 目录，或者写清下载和校验方式
- 提供一个最小验证样例，确认安装后能跑通

这样换一台机器时，用户不需要猜“这个脚本到底依赖什么”。他只需要按固定命令跑。

再往下，沙箱也要更像 Docker。这里不要求一定用 Docker，也不要求沙箱只内置某个固定 Python 版本。更合理的做法是沙箱内置 `uv` 这类环境管理工具，让不同 skill 按自己的配置拉起对应的 Python 版本和依赖。

沙箱需要提供的是可复现能力：稳定的系统依赖、稳定的文件系统约定、明确的网络和权限模型。skill 运行时应该知道自己能读哪里、能写哪里、能不能联网、有没有浏览器、有没有字体。Python 版本和包依赖则应该由 skill 自己声明，再交给 `uv` 这类工具复现。

OpenAI 给了一套 skill 格式规范，但没有规定 Python 工程应该怎么配置，也没有规定依赖、锁文件、验证命令这些工程化内容。要让 Python skill 更容易迁移，这一层需要额外补齐。否则格式只管 skill 怎么写，不管脚本怎么稳定跑。

## 别承诺完整可移植

业务效果很难完全复制。模型和 agent harness 都会变。一个 skill 在某个客户端里表现稳定，换到另一个客户端可能就不一样。问题不一定出在 skill 本身，也可能是执行它的系统换了。

能控制的是运行环境和工程结构。依赖锁住，命令固定，样例能跑，边界写明。

不能控制的是模型能力和 harness 行为。这里不要过度承诺，只能给兼容性说明和测试结果。

我对 skill 可移植性的判断也很简单：它得像一个能被别人安装、运行、验证的小工程。至于在不同 agent 产品里业务效果是否一致，那取决于模型和 harness。skill 作者能做的是降低不确定性，不能消灭不确定性。

把 skill 当工程写，它才有机会被迁移。只把它当提示词写，它大概率只能停在作者自己的工作流里。
