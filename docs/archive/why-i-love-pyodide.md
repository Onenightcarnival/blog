# 为什么我很喜欢 Pyodide

我在博客里加了个 Python 运行时。读者看到代码块，点一下就能跑，不需要后端，不需要装环境。底下就是 Pyodide——把 CPython 编译成 WebAssembly，直接在浏览器里跑。

后来又用它做了个 playground 页面，在线写 Python 即时执行。整个过程没碰服务器，没起 Docker，没写一行后端代码。

用完之后我觉得这东西被低估了。

## 沙箱选型的惯性

一说到"跑不可信的代码"，大家的本能反应是去找一套沙箱基建——Docker 容器、Firecracker microVM、E2B 这种沙箱即服务。每一种都意味着一堆基础设施：镜像管理、资源调度、冷启动优化、容器编排。思路都是"怎么专门围一个笼子"。

这种惯性来自传统后端——代码跑在你服务器上，所以你得在你服务器上做隔离。但 Python 代码其实不一定非要跑在你的后端上。

Pyodide 把这事翻了个面。它跑在 JS 能跑的任何地方——用户的浏览器、你自己的 Node.js、甚至云端的 headless 浏览器。用户写个死循环顶多卡一个 tab 或者一个短命进程，不需要你操心隔离策略，也不需要你造一层新的基建。

不是"更好的沙箱"，是"压根不用自己造沙箱"。

## 后端工程的惯性

用传统沙箱方案跑 Python，你要操心的事一长串：冷启动多久、资源怎么分、并发怎么扩、出了问题怎么查、账单怎么控。每一项单独看都不难，凑到一起就是一整套后端工程。

Pyodide 不一样。用户运行浏览器的设备就是算力，你的服务器什么都不用干。一万个人同时跑代码，你的后端负载是零——算力问题被分散到了每一个终端上。

它的定位就是轻量场景下刚好够用。真正的重活——GPU 运算、完整 Linux 环境、长时间任务——还是得用容器。但绝大多数人要跑的 Python 根本用不到这些，我们只是习惯性地先上了重方案。

就像我之前写的，大多数 LLM 应用不需要框架——同样，大多数跑 Python 的场景，一个 Pyodide 就够了。

好用的工具不一定是最强的，而是最轻的那个刚好够用的。
